# Requirements v1.1: GitHub Monitoring

**Milestone:** v1.1 GitHub Monitoring
**Archived:** 2026-03-23
**Status:** All requirements verified complete

## GitHub Monitoring Requirements

| ID | Requirement | Phase | Status |
|----|-------------|-------|--------|
| GH-01 | User can add a GitHub repository URL to monitor | Phase 4 | Verified |
| GH-02 | System fetches release information using GitHub API (tag_name, body, published_at, html_url) | Phase 4 | Verified |
| GH-03 | System supports GitHub token authentication via environment variable (GITHUB_TOKEN) | Phase 4 | Verified |
| GH-04 | System handles GitHub API rate limits gracefully (60 req/hour unauthenticated, 5000 req/hour with token) | Phase 4 | Verified |
| GH-05 | System detects changelog files (CHANGELOG.md, HISTORY.md, CHANGES.md) via raw.githubusercontent.com | Phase 5 | Verified |
| GH-06 | System scrapes changelog content and stores as article | Phase 5 | Verified |
| GH-07 | New releases and changelog changes are displayed in unified format | Phase 6 | Verified |
| GH-08 | System reuses existing refresh mechanism (fetch --all includes GitHub sources) | Phase 6 | Verified |

## Outcome Summary

All 8 GitHub monitoring requirements (GH-01 through GH-08) and 10 tagging requirements (TAG-01 through TAG-10) were implemented and verified:
- **GH-01–GH-04**: Core GitHub API client with Bearer token auth, rate limit handling
- **GH-05–GH-06**: Changelog detection and scraping via raw.githubusercontent.com
- **GH-07–GH-08**: Unified display and refresh integration
- **TAG-01–TAG-10**: Article tagging system with manual tagging, keyword/regex rules, and AI clustering
