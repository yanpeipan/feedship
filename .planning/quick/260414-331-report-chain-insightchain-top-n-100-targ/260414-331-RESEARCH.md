## User Constraints (from CONTEXT.md)

### Locked Decisions

- 1 cluster (topN articles) generates n topics (by-demand), placed in cluster.children
- Each topic references articles via source_indices (1-based numbering from Cluster.articles)
- TLDRChain is fully deleted
- InsightChain internally generates cluster.summary (single-sentence) + cluster.children (Topic nodes)
- Topic count is by-demand (only topics worth deep-diving get generated), 1 to many
- Insight content: 2-4 sentence coherent paragraph (not list format, not single sentence)
- Minimum 2 articles to generate insight; clusters with fewer articles still get cluster.summary
- Title and content language: target_lang (not bilingual)

## Architecture Patterns

### InsightChain (replace TLDRChain)

Structure mirrors TLDRChain with these differences:

```
BatchClassifyChain(tag_list, target_lang)
  | BuildReportDataChain(heading_tree, target_lang)
  | InsightChain(top_n=100, target_lang=target_lang)  # replaces TLDRChain
```

**InsightChain responsibilities:**
1. Collect all clusters recursively (same as TLDRChain)
2. Filter clusters with >= 2 articles
3. Sort articles by quality_weight desc, take top_n
4. Batch invoke LLM with structured output (same concurrency pattern)
5. Populate cluster.summary (one-sentence TLDR) AND cluster.children (Topic nodes)

**Article presentation to LLM (similar to TLDRChain):**
```
Entity 1 ({cluster.name}):
  [1] article_title_or_translation
  [2] article_title_or_translation
  ...
```

**LLM prompt requirements:**
- Dual-perspective (CEO + AI Analyst)
- Generate n topics by demand (not fixed count)
- Each topic: title, summary (one sentence), multiple insights
- Each insight: title, content (2-4 sentences), source_indices
- Write in target_lang
- Only generate topics worth deep-diving (at least 2 articles to synthesize)

### Variable-Length List Handling

Topic count is variable (by-demand), not fixed. This is supported natively by `.with_structured_output()` — Pydantic's `List[Topic]` handles variable counts without any special configuration. [VERIFIED: Pydantic BaseModel with List fields accepts variable-length arrays from LangChain's with_structured_output]

### Source Index Tracking

Articles sorted by quality_weight desc before numbering. Source indices (1-based) are computed before filtering to top_n. If fewer than top_n articles exist, all are used and numbered accordingly.

Example:
```python
sorted_articles = sorted(cluster.articles, key=lambda a: a.quality_weight, reverse=True)[:self.top_n]
# index i+1 in sorted_articles → source_indices value i+1
```

## Common Pitfalls

### Pitfall 1: Structured output with variable-length lists

**What goes wrong:** LLM may return fewer topics than expected, or occasionally over-generate. Pydantic validation handles this automatically.

**How to avoid:** Use `List[Topic]` in TopicInsightOutput — Pydantic accepts any list length. Trust the model's judgment on topic count.

**Validation:** Log topic count per cluster for quality monitoring.

### Pitfall 2: Empty or single-article clusters

**What goes wrong:** Clusters with 0-1 articles passed to InsightChain.

**How to avoid:** Filter clusters with `len(cluster.articles) < 2` — generate cluster.summary only (via a separate minimal LLM call or skip). The CONTEXT.md specifies: "少于2篇的Cluster不生成Topic/Insight，但cluster.summary仍需生成."

**Implementation:** InsightChain should either skip topic generation for small clusters OR call a separate lightweight chain for cluster.summary. The simplest approach: still call the full chain but instruct the LLM to return empty topics list and only fill summary.

### Pitfall 3: source_indices mismatch after top_n slicing

**What goes wrong:** Indices computed before slicing but LLM sees only top_n articles.

**How to avoid:** Present ALL sorted articles to LLM (not sliced), let LLM reference indices. At invocation time, slice AFTER computing indices.

```python
# WRONG:
presented = sorted_articles[:self.top_n]  # indices become 1-50, but LLM sees 50 articles

# CORRECT:
presented = sorted_articles  # LLM sees all, references by index
# But for very large clusters (>100), still slice to avoid token bloat
# So: present top_n to LLM, indices are 1-based within that window
```

Per CONTEXT decision: top_n=100 means each cluster presents at most 100 articles to the LLM. Indices are 1-based within that presented window.

### Pitfall 4: JSON parse failure in structured output

**What goes wrong:** Model outputs malformed JSON.

**How to avoid:** LLMWrapper already wraps with retry (2 attempts) and JSONSchemaValidationError handling. This is inherited by InsightChain automatically.

## Code Example: New Output Models (in output_models.py)

```python
class Insight(BaseModel):
    title: str = Field(description="Insight subtitle")
    content: str = Field(description="2-4 sentence coherent paragraph")
    source_indices: list[int] = Field(
        description="1-based article indices from the presented list"
    )

class Topic(BaseModel):
    topic_id: str = Field(description="Topic identifier, e.g. Topic_01")
    title: str = Field(description="Topic title in target language")
    summary: str = Field(description="One-sentence deep insight")
    insights: list[Insight] = Field(description="Multiple insights for this topic")

class TopicInsightOutput(BaseModel):
    topics: list[Topic] = Field(description="Variable-length list of topics")
```

## Chain Factory Pattern (in chains.py)

```python
INSIGHT_PROMPT = ChatPromptTemplate.from_messages([
    ("system",
     "You are a senior news analyst with CEO + AI/Technology analyst dual perspectives.\n"
     "Each cluster has multiple related articles on the same subject.\n"
     "Generate TOPICS worth deep-diving (only if 2+ articles can be synthesized).\n"
     "Each topic: title, summary (one sentence), multiple insights.\n"
     "Each insight: title, content (2-4 sentences), source_indices (1-based).\n"
     "Write in {target_lang}."),
    ("human",
     "Cluster articles (top {top_n}):\n"
     "{article_titles}\n\n"
     "Return JSON with 'topics' array.")
])

def get_insight_chain() -> Runnable:
    return INSIGHT_PROMPT | LLMWrapper(structured_output=TopicInsightOutput)
```

## Open Questions

1. **Should clusters with < 2 articles call a separate lightweight LLM for summary, or skip LLM entirely?**
   - Options: (a) Skip LLM, leave summary empty or use a generic placeholder; (b) Call a separate one-sentence summary chain
   - Recommendation: Use a simple instruction in the same prompt — "If fewer than 2 articles, only fill cluster.summary"

2. **Should topic_id be deterministic (e.g., "Topic_01") or generated by LLM?**
   - LLM can generate them as "Topic 1", "Topic 2" — normalize to "Topic_01", "Topic_02" in post-processing
