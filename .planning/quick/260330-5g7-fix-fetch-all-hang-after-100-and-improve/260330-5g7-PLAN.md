  <files>src/cli/feed.py</files>
  <action>
    Replace all uvloop.run() calls with asyncio.run() in src/cli/feed.py (lines 194, 371, 380, 397).

    WHY: uvloop.run() on Python 3.13.5 hangs in shutdown_default_executor() which waits up to 300 seconds for executor threads. asyncio.run() does NOT call shutdown_default_executor() - it directly closes the loop, avoiding this hang entirely.

    Changes needed:
    - Line 194: `uvloop.run(discover_feeds(...))` → `asyncio.run(discover_feeds(...))`
    - Line 371: `uvloop.run(fetch_one_async_by_id(...))` → `asyncio.run(fetch_one_async_by_id(...))`
    - Line 380: `uvloop.run(_fetch_with_progress(...))` → `asyncio.run(_fetch_with_progress(...))`
    - Line 397: `uvloop.run(_fetch_with_progress(...))` → `asyncio.run(_fetch_with_progress(...))`

    Keep the asyncio.constants.THREAD_JOIN_TIMEOUT = 10 patch at line 17 for asyncio.run() fallback (it won't hurt, and asyncio.run() doesn't hang anyway).
  </action>
  <verify>grep -n "uvloop\.run\|asyncio\.run" src/cli/feed.py shows only asyncio.run() calls</verify>
  <done>All uvloop.run() replaced with asyncio.run() in feed.py</done>

<task type="auto">
  <name>Task 2: Batch upsert_articles_async into single transaction</name>
  <files>src/storage/sqlite/impl.py</files>
  <action>
    Refactor upsert_articles_async (lines 269-289) to batch all articles into a SINGLE transaction instead of calling store_article_async in a loop.

    Current problem:
    - Each article calls store_article_async which acquires _db_write_lock
    - With 10 concurrent fetches, all compete for the same lock
    - This serializes writes and kills concurrency benefit

    Fix: Replace the loop-based approach with a single batch upsert that:
    1. Acquires _db_write_lock ONCE
    2. Does ALL articles in a single transaction
    3. Releases lock

    Also fix upsert_articles() (lines 246-266) to do a single transaction for all articles (currently also loops).

    Implementation approach:
    - Create a new `upsert_articles_batch(articles)` function that does bulk insert/update
    - upsert_articles_async calls this with the asyncio.Lock held
    - upsert_articles also calls this directly (no lock needed since sync)

    The batch function should:
    - Build a single INSERT statement with multiple VALUES
    - Use PRAGMA journal_mode=WAL (already set in connection)
    - Do one conn.commit() at the end for all articles
  </action>
