# Phase 8.1: Unified Feed Add - Research

**Researched:** 2026-03-23
**Domain:** GitHub blob URL detection + changelog flow integration
**Confidence:** HIGH

## Summary

Phase 8.1 bridges the gap between "feed add accepts any URL" and "GitHub blob URLs work seamlessly." The current `add_feed()` calls `fetch_feed_content()` + `parse_feed()` which fails (403/422) for GitHub blob HTML pages. The fix: detect GitHub blob URL at the start of `add_feed()` and route to changelog flow instead.

Key findings:
1. `is_github_blob_url()` exists in `src/crawl.py` - detects `github.com/{owner}/{repo}/blob/{branch}/{path}`
2. Changelog functions (`detect_changelog_file()`, `fetch_changelog_content()`, `store_changelog_as_article()`) exist in `src/github.py`
3. `store_changelog_as_article()` requires a `repo_id` linking to `github_repos` table
4. `refresh_feed()` in `src/feeds.py` handles feed refreshs but does not detect GitHub URLs
5. **Critical design gap**: GitHub blob URL feeds need a `github_repos` entry to store changelogs, but `add_feed()` does not create one

### Pattern 1: URL Type Dispatch
**What:** At the start of `add_feed()`, detect URL type and branch to different handlers.
**When to use:** When a single function must handle multiple URL types differently.
**Example:**
```python
def add_feed(url: str) -> Feed:
    # D-03: Detect GitHub blob URL BEFORE fetching as RSS
    github_blob = is_github_blob_url(url)
    if github_blob:
        return add_github_blob_feed(url, github_blob)  # New helper

    # Original RSS/Atom path...
    return add_rss_feed(url)
```

### Pattern 2: Repo-Aware Changelog Storage
**What:** `store_changelog_as_article()` requires a `github_repos` entry. GitHub blob URL feeds must ensure this entry exists.
**When to use:** When GitHub changelog content needs a repo context for storage.
**Example:**
```python
def add_github_blob_feed(url: str, github_blob: tuple) -> Feed:
    owner, repo, branch, path = github_blob

    # Ensure github_repo exists (get or create)
    github_repo = get_or_create_github_repo(owner, repo)

    # Detect and fetch changelog...
    # Store as article with github_repo.id
    # Create feed entry with GitHub URL as url field
```

### Anti-Patterns to Avoid
- **Calling feedparser on GitHub blob URLs:** Causes 403/422 errors. The fix is URL type detection before fetch.
- **Storing changelog without github_repos entry:** `store_changelog_as_article()` requires `repo_id` FK. Must create/link to github_repos first.
- **Silent failures on changelog not found:** D-07 requires explicit error when no changelog detected.

## Common Pitfalls

### Pitfall 1: Forgetting to create github_repos entry
**What goes wrong:** `store_changelog_as_article()` fails with FK constraint error or returns None.
**Why it happens:** `add_feed()` only creates a `feeds` entry, but changelog storage needs `github_repos`.
**How to avoid:** In `add_github_blob_feed()`, ensure `get_or_create_github_repo()` is called before `store_changelog_as_article()`.
**Warning signs:** `IntegrityError: FOREIGN KEY constraint failed` when adding GitHub blob URL feed.

### Pitfall 2: refresh_feed() not handling GitHub URL feeds
**What goes wrong:** `fetch --all` calls `refresh_feed()` which tries to parse GitHub blob page as RSS, causing errors.
**Why it happens:** `refresh_feed()` doesn't check if feed URL is a GitHub blob URL.
**How to avoid:** In `refresh_feed()`, detect GitHub blob URL and call `refresh_changelog()` instead.
**Warning signs:** 403/422 errors in fetch output for GitHub URL feeds.

### Pitfall 3: Explicit blob URL to non-changelog file fails D-07
**What goes wrong:** User adds `README.md` blob URL, `detect_changelog_file()` returns None, feed not added.
**Why it happens:** `detect_changelog_file()` only searches for CHANGELOG/HISTORY/CHANGES files, not README.
**How to avoid:** For explicit blob URLs, fetch the specified file directly via `fetch_changelog_content()` rather than relying on `detect_changelog_file()`. Only fall back to detection if the explicit path doesn't work.
**Warning signs:** `feed add` fails for README.md URLs.

def is_github_blob_url(url: str) -> Optional[tuple[str, str, str, str]]:
    """Detect if URL is a GitHub blob URL.
    Returns:
        Tuple of (owner, repo, branch, path) if match, None otherwise.
    """
    match = GITHUB_BLOB_PATTERN.match(url)
    if match:
        return match.group(1), match.group(2), match.group(3), match.group(4)
    return None

## Open Questions

1. **What if user adds blob URL but repo has no changelog?**
   - What we know: D-07 says "no changelog detected -> error, don't add feed"
   - What's unclear: Should `detect_changelog_file()` be skipped for explicit blob URLs?
   - Recommendation: For explicit blob URLs, fetch the file directly. Only use `detect_changelog_file()` for repo-level additions.

2. **Should GitHub blob URL feeds create a github_repos entry?**
   - What we know: `store_changelog_as_article()` requires `repo_id` FK to `github_repos`
   - What's unclear: Is this coupling desirable? What if user just wants to track README changes?
   - Recommendation: Yes, create github_repos entry (or get existing) so `refresh_changelog()` works. This is consistent with Phase 5 design.

3. **Multiple blob URL feeds for same repo - multiple articles?**
   - What we know: `store_changelog_as_article()` uses `guid = f"changelog:{repo_name}:{filename}"`. Same filename = same guid = overwrite.
   - What's unclear: If user adds CHANGELOG.md and then README.md blob URLs for same repo, do they get 1 article or 2?
   - Recommendation: Currently 1 article (last one wins). This is a limitation but out of scope for gap closure.
